# PRD: DeckDeckDeck Android Native MVP

- 문서 버전: v0.1
- 작성일: 2026-06-27
- 대상 플랫폼: Android
- 구현 방식: Kotlin Native Android 앱
- 원본 기준: Windows C#/WPF DeckDeckDeck의 핵심 UX
- MVP 범위: 오버레이 실행, 문구 저장/복사, 웹사이트 열기
- 제외 범위: 핫키, 파일 실행, 미디어 제어, 터미널 명령, 클라우드 동기화

---

## 1. 문서 목적

이 문서는 Windows C#/WPF 프로그램인 DeckDeckDeck의 핵심 사용 경험을 Android Kotlin native 앱으로 옮기기 위한 제품 요구사항을 정의한다.

현재 단계의 목적은 원본 앱의 모든 기능을 재현하는 것이 아니다.  
사용자가 실제로 편리함을 느낀 핵심 경험, 즉 **자주 쓰는 항목을 저장해두고 필요할 때 빠르게 실행하는 카드형 개인 런처 경험**을 모바일 환경에 맞게 축소 구현한다.

이번 Android MVP는 다음 실행 경험만 포함한다.

1. 앱 실행 시 직전 화면 위에 오버레이 실행 패널 표시
2. 문구 저장 후 빠른 복사
3. 웹사이트 주소 저장 후 빠른 열기

핫키 기능은 포함하지 않는다.

---

## 2. 배경과 계기

사용자는 기존 Windows DeckDeckDeck을 사용하면서 다음과 같은 편의성을 느꼈다.

1. 자주 쓰는 문구를 저장해두고 반복해서 사용할 수 있음
2. 자주 수행하는 루틴을 카드처럼 등록하고 빠르게 실행할 수 있음
3. 사용자가 직접 구성한 개인용 런처처럼 동작함
4. 서버, 계정, 클라우드 없이 개인 기기 안에서 관리할 수 있음

사용자는 이 경험을 휴대폰에서도 사용하고 싶어 한다.

초기 Android MVP에서는 모바일 제약과 구현 복잡도를 고려해, 자동화 루틴 전체가 아니라 **문구 재사용**과 **웹사이트 실행**만 먼저 구현한다.

---

## 3. 제품 한 줄 정의

**DeckDeckDeck Android는 자주 쓰는 문구와 자주 여는 웹사이트를 카드로 저장해두고, 직전 화면 위에 뜨는 오버레이에서 탭 한 번으로 복사하거나 여는 모바일 개인 런처 앱이다.**

---

## 4. 제품 목표

### 4.1 핵심 목표

1. 사용자가 반복해서 쓰는 문구를 쉽게 저장하고 빠르게 복사할 수 있게 한다.
2. 사용자가 자주 여는 웹사이트를 카드로 저장하고 빠르게 열 수 있게 한다.
3. 카테고리와 카드 기반 UI를 통해 사용자가 자신의 사용 패턴에 맞게 정리할 수 있게 한다.
4. 서버, 계정, 클라우드 동기화 없이 로컬 우선 앱으로 동작하게 한다.
5. 원본 Windows 앱의 “Soft Keycap Card” 감각을 모바일에 맞게 유지한다.

### 4.2 비목표

이번 MVP에서는 다음을 구현하지 않는다.

| 항목 | 제외 사유 |
|---|---|
| 핫키 | Android 모바일 UX와 맞지 않고, 시스템 제약이 큼 |
| 현재 커서 직접 붙여넣기 | 오버레이만으로는 다른 앱의 현재 입력 위치에 직접 입력할 수 없으므로, MVP에서는 클립보드 복사로 확정 |
| 파일 실행 | 모바일 MVP 범위 밖 |
| 미디어 제어 | 모바일 MVP 범위 밖 |
| 터미널 명령 | 모바일 MVP 범위 밖 |
| Galaxy Buds 연결/해제 자동화 | 추후 Android Intent/Settings 연동 기능 후보 |
| 클라우드 동기화 | 로컬 우선 MVP 원칙 유지 |
| 계정 시스템 | 개인용 로컬 도구로 시작 |
| 백업/복원 | 초기 MVP 이후 데이터 안정성 단계에서 검토 |

## 5. 대상 사용자

### 5.1 주요 사용자

반복 입력, 자주 여는 링크, 개인 루틴을 직접 구성해서 쓰는 사용자를 대상으로 한다.

예시:

1. 자주 쓰는 답장 문구를 저장해두고 싶은 사용자
2. ChatGPT, Claude, Gemini 등에 반복 입력하는 프롬프트를 저장해두고 싶은 사용자
3. 자주 여는 사이트를 목적별로 정리하고 싶은 사용자
4. 휴대폰에서 개인용 미니 런처를 구성하고 싶은 사용자
5. 서버나 계정 없이 로컬에만 데이터를 저장하고 싶은 사용자

### 5.2 대표 사용 시나리오

#### 시나리오 A: 반복 문구 사용

사용자는 자주 쓰는 문구를 카드로 저장한다.  
이후 오버레이에서 카드 탭 한 번으로 저장된 문구를 클립보드에 복사하고, 사용자가 원래 앱의 현재 입력 위치에 수동으로 붙여넣는다.

예시 문구:

- 자기소개
- 계좌 안내
- 회의 요약 프롬프트
- 코드 리뷰 요청 템플릿
- 반복적인 메신저 답장
- 고객 응대 문구

#### 시나리오 B: 자주 여는 웹사이트 실행

사용자는 웹사이트 주소를 카드로 저장한다.  
이후 카드 탭 한 번으로 기본 브라우저 또는 Custom Tab에서 해당 웹사이트를 연다.

예시 웹사이트:

- ChatGPT
- GitHub
- Notion
- Gmail
- Google Drive
- 회사 내부 페이지
- 자주 보는 문서 링크

## 8. 화면 구성

## 8.1 오버레이 실행 화면

### 목적

앱 실행 시 일반 Deck 앱 화면을 보여주지 않고, 직전에 사용하던 화면 위에 카테고리/카드 실행 패널을 띄운다.

### 주요 요소

| 요소 | 설명 |
|---|---|
| 반투명 배경 | 직전 화면이 살짝 보이게 하여 현재 작업 맥락을 유지 |
| 상단 영역 | 현재 위치, 뒤로가기, 설정 아이콘 |
| 카테고리 그리드 | 카테고리 카드 목록 |
| 실행 카드 그리드 | 선택한 카테고리의 카드 목록 |
| 상태 피드백 | 복사 성공, URL 열기 실패 등 Toast/Snackbar |

### 사용자 행동

| 행동 | 결과 |
|---|---|
| 앱 실행 | 직전 화면 위에 오버레이 표시 |
| 카테고리 카드 탭 | 해당 카테고리의 카드 목록 표시 |
| 문구 카드 탭 | 저장된 문구를 클립보드에 복사 후 오버레이 종료 |
| 웹사이트 카드 탭 | 웹사이트를 열고 오버레이 종료 |
| 뒤로가기 | 카드 목록에서는 카테고리 목록으로, 카테고리 목록에서는 오버레이 종료 |
| 설정 아이콘 탭 | 일반 설정/관리 화면으로 이동 |

### UX 원칙

1. 앱 실행 시 일반 Deck 앱 화면이 먼저 보이면 안 된다.
2. 오버레이 뒤에는 직전에 사용하던 화면이 보여야 한다.
3. 작은 플로팅 핸들은 사용하지 않는다.
4. 카드 실행 후에는 오버레이와 관련 서비스가 정리되어야 한다.
5. 카드가 많아져도 스크롤로 접근 가능해야 한다.
6. 모바일에서는 원본 텐키 배열을 그대로 강제하지 않는다.
7. 대신 고정된 카드 그리드 감각은 유지한다.

## 8.2 오버레이 카테고리 상세 화면

### 목적

선택한 카테고리 안의 실행 카드를 오버레이 안에서 보여주고, 탭 한 번으로 실행하게 한다.

### 주요 요소

| 요소 | 설명 |
|---|---|
| 상단 영역 | 뒤로가기, 카테고리 이름 |
| 실행 카드 그리드 | 문구/웹사이트 카드 목록 |
| 카드 타입 표시 | 문구/웹사이트 구분 아이콘 또는 배지 |
| 상태 피드백 | Snackbar/Toast |

### 사용자 행동

| 행동 | 결과 |
|---|---|
| 문구 카드 탭 | 문구 클립보드 복사 후 오버레이 종료 |
| 웹사이트 카드 탭 | 웹사이트 열기 후 오버레이 종료 |
| 뒤로가기 | 오버레이 카테고리 목록으로 이동 |

### UX 원칙

1. 카드 탭은 즉시 실행으로 동작한다.
2. 오버레이는 실행 전용으로 두고, 편집은 설정/관리 화면에서만 제공한다.
3. 카드의 실행 타입이 시각적으로 구분되어야 한다.
4. 실행 결과는 짧고 명확하게 표시한다.
5. 실패 시에는 오버레이를 유지하고 짧은 오류 메시지를 표시한다.

## 8.3 실행 카드 편집 화면

### 목적

문구 카드 또는 웹사이트 카드를 생성/수정한다.

### 공통 필드

| 필드 | 필수 여부 | 설명 |
|---|---:|---|
| 슬롯 명 | 필수 | 카드 제목 |
| 실행 타입 | 필수 | 문구 복사 또는 웹 주소 열기 |
| 설명 | 선택 | 카드 설명 |
| 이미지 | 선택 | 카드 썸네일 |
| 이 카드 사용 | 선택 | 활성/비활성 |

### 문구 복사 타입 필드

| 필드 | 필수 여부 | 설명 |
|---|---:|---|
| 복사할 문구 | 필수 | 클립보드에 복사할 텍스트 |

문구 입력 필드는 여러 줄 입력을 지원해야 한다.

지원해야 하는 내용:

- 줄바꿈
- Markdown
- 코드 블록
- 긴 텍스트
- 이모지
- 한글/영문 혼합 텍스트

### 웹 주소 열기 타입 필드

| 필드 | 필수 여부 | 설명 |
|---|---:|---|
| 열 웹페이지 주소 | 필수 | 열 URL |

URL 입력 필드는 저장 시 자동 정규화를 수행한다.

예시:

| 입력값 | 저장값 |
|---|---|
| `google.com` | `https://google.com` |
| `github.com/asambrive4-rgb/deckdeckdeck` | `https://github.com/asambrive4-rgb/deckdeckdeck` |
| `https://chatgpt.com` | `https://chatgpt.com` |

### 하단 액션

| 버튼 | 설명 |
|---|---|
| 삭제 | 기존 카드 삭제 |
| 취소 | 변경사항 폐기 |
| 저장 | 검증 후 저장 |

## 9. 핵심 사용자 흐름

## 9.1 문구 카드 생성 흐름

```text
설정/관리 화면
→ 카테고리 선택
→ 빈 카드 탭
→ 실행 타입: 문구 복사 선택
→ 슬롯 명 입력
→ 복사할 문구 입력
```

## 10. 다음 구현 구획: 오버레이 우선 실행 + 문구 복사

### 10.1 목표

다음 구현 구획은 앱을 실행했을 때 일반 Deck 앱 화면을 먼저 보여주지 않고, 직전에 사용하던 화면 위에 Deck 오버레이 실행 패널을 띄우는 것을 목표로 한다.

문구 카드는 현재 커서 위치에 직접 입력하지 않고, 저장된 문구를 클립보드에 복사한다. 직접 붙여넣기는 IME 또는 접근성 서비스 검토가 필요한 별도 후보로 남긴다.

### 10.2 포함 범위

| 항목 | 설명 |
|---|---|
| 오버레이 권한 확인 | 권한이 없으면 Android 설정 화면으로 안내 |
| 앱 실행 라우팅 | 앱 실행 시 일반 관리 화면 대신 오버레이 표시 |
| 오버레이 카테고리 목록 | 카테고리 카드를 오버레이에 표시 |
| 오버레이 카드 목록 | 선택한 카테고리의 실행 카드를 오버레이에 표시 |
| 설정 진입 | 오버레이 우측 상단 설정 아이콘으로 일반 관리 화면 진입 |
| 설정 복귀 | 설정 화면 뒤로가기 시 오버레이로 복귀 |
| 문구 복사 | 문구 카드 탭 시 클립보드에 복사 |
| 웹사이트 열기 | 웹사이트 카드 탭 시 브라우저 또는 Custom Tab으로 열기 |
| 실행 후 정리 | 카드 실행 후 오버레이 종료, Activity 종료, Overlay Service 정리 |

### 10.3 제외 범위

| 항목 | 제외 이유 |
|---|---|
| 작은 플로팅 핸들 | 사용자는 앱 실행 자체를 오버레이 호출 트리거로 사용하기를 원함 |
| 현재 커서 직접 붙여넣기 | 오버레이만으로는 불가능하므로 MVP에서는 복사로 확정 |
| IME 구현 | 별도 검토가 필요한 큰 기능 |
| 접근성 서비스 구현 | 권한과 동작 안정성 검토가 필요한 큰 기능 |
| 카드 수정/삭제 고도화 | 이번 구획은 실행 UX 전환이 목표 |
| 이미지 선택 | 이번 구획 범위 밖 |

### 10.4 UX 흐름

```text
사용자가 Deck 앱 실행
→ Deck 일반 앱 화면은 보이지 않음
→ 직전에 쓰던 화면 위에 Deck 오버레이 표시
→ 카테고리 선택
→ 카드 선택
→ 문구 복사 또는 웹사이트 열기
→ 오버레이 종료
→ Activity 종료 및 Overlay Service 정리
```

설정 화면 흐름은 다음과 같다.

```text
오버레이 표시
→ 우측 상단 설정 아이콘 탭
→ 일반 설정/관리 화면 진입
→ 뒤로가기
→ 오버레이로 복귀
```

### 10.5 성공 기준

1. 앱 실행 직후 Deck 일반 앱 화면이 먼저 보이지 않는다.
2. 오버레이 뒤에는 직전에 사용하던 화면이 보여야 한다.
3. 오버레이에서 카테고리와 카드를 탐색할 수 있다.
4. 문구 카드 탭 시 저장된 문구가 클립보드에 복사된다.
5. 웹사이트 카드 탭 시 URL이 열린다.
6. 카드 실행 후 오버레이, Activity, Overlay Service가 정리된다.
7. 설정 아이콘으로 관리 화면에 진입하고, 뒤로가기 시 오버레이로 복귀한다.
8. 오버레이 권한이 없으면 사용자가 권한 설정으로 이동할 수 있다.

### 10.6 테스트 계획

```powershell
.\gradlew.bat :app:testDebugUnitTest
.\gradlew.bat :app:assembleDebug
```

추가로 실제 기기에서 다음을 수동 확인한다.

1. 삼성 One Hand Operation으로 앱 실행 시 오버레이가 먼저 뜨는지 확인한다.
2. 앱 아이콘으로 실행해도 일반 Deck 화면이 먼저 보이지 않는지 확인한다.
3. 오버레이 권한이 꺼져 있을 때 안내 흐름이 동작하는지 확인한다.
4. 문구 카드 실행 후 클립보드에 복사되는지 확인한다.
5. 카드 실행 후 원래 화면으로 돌아가는지 확인한다.

## 12. URL 정규화 규칙

### 12.1 기본 규칙

1. 앞뒤 공백을 제거한다.
2. 문자열 내부 공백이 있으면 실패 처리한다.
3. `://`가 없으면 앞에 `https://`를 붙인다.
4. `http`와 `https`만 허용한다.
5. host가 비어 있으면 실패 처리한다.
6. 일반 도메인은 `.`을 포함해야 한다.
7. `localhost`와 IP 주소는 허용할 수 있다.

### 12.2 예시

| 입력 | 결과 |
|---|---|
| `google.com` | 성공: `https://google.com` |
| `https://google.com` | 성공: `https://google.com` |
| `http://localhost:3000` | 성공 |
| `192.168.0.1` | 성공: `https://192.168.0.1` |
| `hello world.com` | 실패 |
| `ftp://example.com` | 실패 |
| `example` | 실패 |

## 13. 문구 복사 규칙

### 13.1 기본 동작

문구 카드 실행 시 앱은 저장된 문구를 클립보드에 복사한다.

오버레이만으로는 다른 앱의 현재 커서 위치에 직접 입력할 수 없으므로, MVP에서는 현재 커서 직접 붙여넣기를 목표로 하지 않는다. 직접 입력은 IME 또는 접근성 서비스 검토가 필요한 향후 후보로 남긴다.

### 13.2 UX 피드백

복사 성공 시 Snackbar 또는 Toast를 표시한다.

```text
복사했습니다
```

### 13.3 주의사항

클립보드는 Android 시스템의 공유 영역이므로, 사용자는 복사 후 원래 앱에서 수동으로 붙여넣어야 한다.
민감한 문구를 복사할 수 있으므로 향후에는 클립보드 자동 삭제, 민감 문구 표시, 복사 이력 노출 여부를 별도로 검토한다.

## 15. 권한과 플랫폼 제약

### 15.1 필요한 권한

오버레이 실행을 위해 시스템 오버레이 권한이 필요하다.

| 기능 | 권한 |
|---|---|
| 오버레이 표시 | `SYSTEM_ALERT_WINDOW` 권한 필요 |
| 문구 복사 | 별도 런타임 권한 없음 |
| 웹사이트 열기 | 별도 런타임 권한 없음 |
| 이미지 선택 | Android 버전에 따라 Photo Picker 또는 미디어 접근 권한 |
| 로컬 DB 저장 | 별도 권한 없음 |

### 15.2 오버레이와 문구 복사 구현

Android는 보안상 앱이 다른 앱의 입력창에 임의로 텍스트를 자동 입력하는 것을 일반 앱 권한으로 허용하지 않는다.  
따라서 이번 MVP의 문구 실행은 클립보드 복사로 확정한다.

앱 실행 시 직전 화면 위에 Deck 패널을 띄우기 위해 오버레이 권한을 사용한다. 권한이 없으면 사용자를 Android 설정 화면으로 안내한다.

## 19. MVP 수용 기준

### 19.1 카테고리

- [ ] 사용자는 카테고리를 생성할 수 있다.
- [ ] 사용자는 카테고리 이름을 수정할 수 있다.
- [ ] 사용자는 카테고리를 삭제할 수 있다.
- [ ] 오버레이와 설정/관리 화면에서 카테고리 카드가 표시된다.
- [ ] 카테고리 탭 시 해당 카테고리 상세 화면으로 이동한다.

### 19.2 문구 카드

- [ ] 사용자는 문구 카드를 생성할 수 있다.
- [ ] 사용자는 긴 문구, 줄바꿈, Markdown을 저장할 수 있다.
- [ ] 문구 카드 탭 시 저장된 문구가 클립보드에 복사된다.
- [ ] 복사 성공 시 피드백이 표시된다.
- [ ] 제목 또는 문구가 비어 있으면 저장되지 않는다.

### 19.3 웹사이트 카드

- [ ] 사용자는 웹사이트 카드를 생성할 수 있다.
- [ ] `example.com` 입력 시 `https://example.com`으로 저장된다.
- [ ] `http`와 `https` 외 스킴은 저장되지 않는다.
- [ ] 웹사이트 카드 탭 시 브라우저 또는 Custom Tab이 열린다.
- [ ] URL 형식이 잘못되면 오류 메시지가 표시된다.

### 19.4 편집 UX

- [ ] 빈 카드 탭 시 새 카드 생성 화면으로 이동한다.
- [ ] 기존 카드 길게 누르기 또는 메뉴로 편집할 수 있다.
- [ ] 기존 카드 삭제가 가능하다.
- [ ] 저장/취소/삭제 액션이 명확하게 구분된다.

### 19.5 로컬 저장

## 22. 최종 MVP 요약

Android MVP는 다음 한 문장으로 정의한다.

**카테고리별로 자주 쓰는 문구와 자주 여는 웹사이트를 카드로 저장하고, 직전 화면 위에 뜨는 오버레이에서 탭 한 번으로 복사하거나 여는 로컬 개인 런처 앱.**

이번 MVP에서 반드시 지켜야 할 기준은 다음과 같다.

1. 핫키는 넣지 않는다.
2. 실행 타입은 문구 복사와 웹 주소 열기만 둔다.
3. 문구 실행의 MVP 목표는 클립보드 복사다.
4. 현재 커서 직접 붙여넣기는 IME 또는 접근성 서비스 검토 전까지 MVP에 넣지 않는다.
5. 웹사이트 실행은 URL 정규화 후 브라우저 또는 Custom Tab으로 연다.
6. 데이터는 로컬에 저장한다.
7. 카드는 빠르게 알아보고 빠르게 실행할 수 있어야 한다.
8. 원본 DeckDeckDeck의 카드형 런처 감각은 유지한다.
