# File Naming Guidelines

파일 이름은 코드의 책임을 빠르게 읽기 위한 약속이다.  
같은 역할을 하는 파일은 같은 접미사를 사용하고, 접미사만 봐도 파일의 책임을 짐작할 수 있어야 한다.

## 원칙

- 같은 역할을 하는 파일은 같은 접미사를 쓴다.
- 접미사는 파일의 책임을 드러내야 한다.
- 접미사만 보고 파일이 어느 계층에서 어떤 일을 하는지 짐작할 수 있어야 한다.
- 도메인 개념 자체가 파일의 책임을 충분히 설명한다면 불필요한 접미사를 붙이지 않는다.
- 기존 패키지나 폴더에 이미 자리 잡은 이름 규칙이 있다면 먼저 따른다.
- 새 접미사를 만들기 전에 기존 접미사로 표현할 수 있는지 확인한다.
- 이름 규칙은 다음 파일을 추가할 때도 반복 가능해야 한다.

## 행동 지침

- 새 파일을 만들기 전에 같은 역할의 기존 파일 이름을 먼저 찾아본다.
- 같은 역할인데 접미사가 다르면 하나로 통일한다.
- 파일 이름이 너무 넓거나 모호하면 책임을 더 잘 드러내는 접미사로 바꾼다.
- 파일 이름이 도메인 개념과 기술 세부사항을 불필요하게 섞고 있다면 분리하거나 더 단순한 이름을 고른다.
- 접미사 통일만으로 책임이 명확해지지 않으면 파일의 책임 자체가 섞인 것은 아닌지 확인한다.
- 이름 변경 후 import, package, 테스트 이름도 함께 맞춘다.
- 이름 변경 후 가능한 자동 테스트나 빌드로 동작이 유지되는지 확인한다.

## 점검 질문

- 같은 역할의 다른 파일과 접미사가 같은가?
- 접미사만 보고 파일의 책임을 짐작할 수 있는가?
- 이 이름이 특정 구현보다 역할을 먼저 설명하는가?
- 도메인 모델에는 불필요한 기술 접미사를 붙이지 않았는가?
- 다음에 같은 역할의 파일을 추가할 때도 같은 규칙을 적용할 수 있는가?
